# 秘密更新链（隐藏 Boss / 轮回层级 / 深渊回响）设计（2026-03-08）

## 当前项目现状理解

基于当前 `V3.0-beautiful` 分支代码，和这轮需求讨论，我对项目现状的判断如下：

1. `src/ui/scenes/title.ts` 目前还是极简标题页：`开始冒险 / 样式工坊 / 继续冒险`，没有“轮回层级”入口，也没有可承载解锁说明的面板结构。
2. `src/game/meta.ts` 已经承担“跨局长期进度”的职责：解锁、里程碑、隐藏起始包、传承武器都在这里持久化；`src/game/save.ts` 又会把 `metaProfile` 连同存档一起保存，所以“轮回层级解锁”和“通关者摘要”最适合挂在 `metaProfile` 上。
3. `src/game/run.ts` 的 `createRunState(...)` 已经负责把少量元进度注入新开局；这正好符合“轮回层级只影响新开局，不影响已有存档继续”的要求——新 run 锁定本局 `cycleTier`，继续旧档时不改。
4. `src/main.ts` 目前掌控战斗胜负、Boss 结算、转幕、奖励、结果页，是“第三幕 Boss 战后跳转到隐藏线”的最佳接入点。
5. `src/game/events.ts` + `src/ui/scenes/event.ts` 已经有成熟的“标题 + 描述 + 选项按钮”事件页。它还不算 cutscene 系统，但足够承载：
   - 被侵染的鸣谢
   - 门的回声
   - 门仍未尽开
   - 门后之言
6. 当前敌人系统是典型“敌人/Boss 固定意图序列”：`src/game/enemies.ts` 定义意图，`src/game/combat.ts` 执行，不存在玩家卡组 AI。这一点反而非常适合“锻铸者残响像玩家、但不是玩家”的方向。
7. 当前分支里还没有独立的“秘密更新链模块”；这意味着最稳的做法不是散改多个旧系统，而是新增一个小型的 `secret-cycle` 领域模块，把解锁、摘要、隐藏线流程、回响 Boss 生成都收口进去，再由 `main.ts` 和标题页调用。

一句话概括：**现有框架非常适合做“事件驱动 + 元进度驱动 + 固定 Boss 骨架”的秘密更新链，不适合做完整玩家镜像 AI。**

## 目标与非目标

### 目标

- 为标题页增加世界观命名的“轮回层级”选择。
- 在第一轮回保留原始隐藏 Boss 首次发现体验。
- 在第二轮回及深渊回响层引入“双战结构”：
  - 第一战：锻铸者残响
  - 第二战：原隐藏 Boss 真身 / 强化态
- 用“中继承摘要”生成残响 Boss，保留“上一位冒险者黑化”的核心味道。
- 引入“裂隙之刃”作为轮回链象征奖励与隐藏起始武器。
- 保持实现可控，不重写战斗框架。

### 非目标

- 不实现完整玩家卡组镜像。
- 不实现敌人抽牌/出牌系统。
- 不让 Boss 走玩家回合制资源逻辑。
- 不把隐藏线做成完整第四幕大地图。
- 不为了这个需求重写现有 UI 体系。

## 方案比较

### 方案 A：完整玩家镜像 Boss

**做法**
- 记录上一位通关者整套卡组、抽牌顺序、战斗日志。
- 敌人侧直接复用玩家出牌逻辑。

**优点**
- “上一位冒险者黑化”味道最直观。

**缺点**
- 与当前敌人系统完全背道而驰。
- AI、抽牌、资源、卡牌选择都会极大放大复杂度。
- 调试成本极高，平衡几乎不可控。

**结论**
- 明确不推荐；与本次非目标冲突。

### 方案 B：固定 Boss 骨架 + 摘要驱动修正包（推荐）

**做法**
- 只继承“武器 + 最多 2 个附魔 + 2 个机制标签 + 一句墓志铭”。
- 由这些摘要映射到：
  - 武器人格模板
  - 附魔腐化被动
  - 招牌机制模块
- 残响仍然是敌人式固定行动节奏。

**优点**
- 和当前 `EnemyDef + intent` 架构匹配。
- 既能让玩家一眼认出“像上一位冒险者”，又不会变成不可维护的玩家 AI。
- 后续还可以继续扩成更正式的秘密更新。

**缺点**
- 需要补一个小型“残响生成器”和少量敌人被动支持。
- 需要认真设计摘要字段与映射关系。

**结论**
- 最符合本次需求、实现风险最低，推荐采用。

### 方案 C：隐藏线改为完整 Act 4 地图

**做法**
- 第三幕 Boss 后进入一整张隐藏地图，包含事件、商店、营火、两场 Boss。

**优点**
- 叙事空间大。

**缺点**
- 需要改地图、转幕、资源节奏、掉落结构。
- 会把“秘密更新链”变成“新增一整幕”的量级。
- 与“先不要大改”“保守可落地”相反。

**结论**
- 当前阶段不推荐。

## 推荐方案总览

### 核心思路

采用**方案 B：事件驱动的隐藏线 + 中继承残响 Boss + 轮回层级元进度**。

整体上新增一个 `secret-cycle` 领域模块，负责：

- 轮回层级显示名与解锁规则
- 第一轮回 / 第二轮回 / 深渊回响的流程差异
- 通关者摘要提取与回退模板
- 锻铸者残响生成
- 深渊回响词缀 / 变异规则
- 隐藏线事件文案编排

`main.ts` 只保留“何时进入隐藏线 / 何时开始哪一战 / 何时结算奖励”的薄胶水逻辑。

## 一、轮回层级设计

### 层级命名

- `0` -> 第一轮回
- `1` -> 第二轮回·侵影
- `2` -> 深渊回响·1
- `3` -> 深渊回响·2
- `4` -> 深渊回响·3
- 以此类推

### 解锁规则

- 第一轮回默认开放。
- 第一次击败隐藏 Boss：
  - 解锁称号：`超越者`
  - 解锁隐藏起始武器：`裂隙之刃`
  - 解锁轮回：`第二轮回·侵影`
- 完成 `第二轮回·侵影`：解锁 `深渊回响·1`
- 击败 `深渊回响·N`：解锁 `深渊回响·N+1`
- 允许向下选择更低层级。

### 标题页展示规则

这里需要吸收你后续补充的 UI 方向：

- 在**首次击败隐藏 Boss 之前**，标题页不直接展示轮回列表。
- 取而代之显示一个被封印的区域，例如：
  - 标题：`不要进来`
  - 副文案：`门还没有为你打开。？？？？还没有允许你靠近。`
- 在**首次击败隐藏 Boss 之后**，这个区域才展开为“轮回层级”选择面板。
- 未解锁层级灰显，并显示解锁条件：
  - `击败门后的存在`
  - `完成第二轮回·侵影`
  - `击败深渊回响·N`

### 只影响新开局

- 标题页的层级选择只在点击“开始冒险”时写入本局 run。
- 旧存档继续时，不读取新的标题选择，不强制切层。
- 也就是说：
  - `metaProfile` 记录“你解锁到了哪一层”
  - `RunState` 记录“这局是从哪一层开的”

这是最稳的持久化边界。

## 二、隐藏线叙事流程

### 1）首次激活隐藏线（玩家从未击败过隐藏 Boss）

事件顺序定为：

1. 击败第三幕普通最终 Boss
2. **不显示**“门的回声 / 容器识别台词”
3. 显示 `被侵染的鸣谢`
4. 按钮：`把手放上去`
5. 直接进入隐藏 Boss 战
6. 战后显示 `门后之言`
7. 发放奖励：
   - 称号 `超越者`
   - 起始武器 `裂隙之刃`
   - 解锁 `第二轮回·侵影`

这样首轮体验更像“第一次真正发现门后真相”，不会被“普通最终 Boss 识别你”提前剧透。

### 2）第二次及以后进入隐藏线

事件顺序定为：

1. 击败第三幕普通最终 Boss
2. 显示 `门的回声`（普通最终 Boss 的识别台词）
3. 显示重新进入事件：
   - 第二次：`侵染回响`
   - 第三次及以后：`回响仍在`
4. 第一战：`锻铸者残响`
5. 战后过场：`门仍未尽开`
6. 第二战：隐藏 Boss 真身 / 强化态
7. 战后显示 `门后之言`
8. 结算本层解锁

### 3）普通最终 Boss 的后续识别台词

这段台词只在**玩家已经至少击败过一次隐藏 Boss**之后出现。

推荐润色文案：

> 就是你了……我已经等了很久。  
> 倒在门前的人太多，能走到这里的人太少。  
> 别把这当成胜利。  
> 我需要的从来不是挑战者。  
> 我需要的，是能替我站在这里的人。  
> 去吧——门会认出你。

语义上保留了“容器 / 守门者 / 继承体”的感觉，不是认输，而是筛选。

## 三、事件文案与展示建议

### A. 首次隐藏线入口：`被侵染的鸣谢`

推荐分段展示文案：

1.
`感谢你玩到这里。`

2.
`如果你正在读这段话，说明你已经把这条路，走到了大多数人不会抵达的地方。`

3.
`我是一个产品经理，最近一直在尝试借助 AI，把脑海里那款想了很久的游戏，一点一点做出来。`

4.
`它还不完整，也还在生长，但你能来到这里，已经说明你真的看见了它。`

5.
`谢谢你愿意体验、包容、停留。`
`也谢谢你，把这条路走得比我最初想象得更远。`

6.
`因为通常不会有人到这里。`
`不会有人继续往下。`
`不会有人听见门后的声音。`

7.
`可总要有一个。`
`？？？？总要有一个。`
`而现在，门已经记住你了。`
`请继续前进。`

按钮建议：`把手放上去`

### B. 第二次进入：`侵染回响`

`你已经来过这里。`
`那些感谢，本不该再说第二次。`
`门在后面，你知道。`
`门也知道你会回来。`
`？？？？已经认得你了。`
`继续前进。`

按钮建议：`再次触门`

### C. 第三次及以后：`回响仍在`

`不必再自我介绍了。`
`你与门之间，早已不需要言语。`
`每一次返回，都会留下些什么。`
`每一次推开，都会失去些什么。`
`？？？？仍在等待。`

按钮建议：`让门打开`

### D. 双战中场：`门仍未尽开`

`你击碎的只是外层回响。`
`真正的门还没有完全开启。`
`再往前一步。`
`让它看清你。`

按钮建议：`继续深入`

### E. 最终战后：`门后之言`

`门不会为弱者开启，也不会为胜者停留。`
`你已越过它一次，便会被它永远记住。`
`带着裂隙离开。`
`或者回来，直到所有回响都只剩你的名字。`

按钮建议：`接受回响`

### 展示形式建议

不建议为了这批文案单独做一套新 cutscene 系统；一期只做下面两点就够：

1. `EventDef` 支持多段文本数组，而不是只有单段 `description`
2. `event` 场景支持一个 `presentation: 'default' | 'abyss'`，给隐藏线事件挂污染纹理、字色和更重的 `？` 演出

这样能显著提升氛围，但不会把 UI 复杂度拉爆。

## 四、通关者摘要结构设计

### 推荐结构

```ts
interface ChampionSummary {
  version: 1
  weaponDefId: string
  enchantments: EnchantmentId[]
  primaryArchetype: EchoArchetypeId | null
  secondaryArchetype: EchoArchetypeId | null
  signatureTraitA: EchoTraitId | null
  signatureTraitB: EchoTraitId | null
  epitaph: string
  clearedCycleTier: number
  usedRiftBlade: boolean
  capturedAt: number
}
```

### 字段解释

- `weaponDefId`
  - 核心字段，决定残响 Boss 的基础人格模板。
- `enchantments[0..2]`
  - 实际使用时只取最多 2 个；存储上可以容忍旧数据脏值，但加载时裁成前 2 个。
- `primaryArchetype / secondaryArchetype`
  - 描述这位通关者的总体走向，例如：
    - `martial_burst`
    - `spell_burst`
    - `burn`
    - `poison`
    - `armor`
    - `charge`
    - `combo`
    - `resource_burst`
    - `lifesteal`
    - `armor_break`
- `signatureTraitA / B`
  - 具体驱动残响技能模块的标签，和 archetype 不必完全同义。
- `epitaph`
  - 一条简短战绩摘要，用于残响事件文案、Boss 副标题、深渊回响 flavor。
- `clearedCycleTier`
  - 这份摘要对应哪一层通关者。
- `usedRiftBlade`
  - 裂隙之刃需要特殊处理时非常有用。

### 为什么这套结构最稳

- 它不依赖完整战斗日志。
- 它不依赖抽牌顺序。
- 它不要求“上一局玩家 AI 可复现”。
- 它足够小，适合放进 `metaProfile` 和存档。
- 它足够强，能驱动“像上一位玩家”的 Boss 观感。

### 摘要提取策略（推荐）

不要从全量战斗日志硬推；一期用**静态推断**即可：

1. **武器**：直接读 `run.equippedWeapon.defId`
2. **附魔**：直接读 `run.equippedWeapon.enchantments.slice(0, 2)`
3. **流派标签**：扫描最终 deck 的 `CardDef.category + effects`
   - `combat` 占比高 + 高伤害卡多 -> `战技爆发`
   - `spell` 占比高 + 法术倍率卡多 -> `法术爆发`
   - `burn` 效果多 -> `灼烧`
   - `poison` 效果多 -> `中毒`
   - `armor` / `conditional_armor` 多 -> `高护甲`
   - `gain_charge` / 法杖系多 -> `蓄能`
   - `multi_damage` / 低费 combat 多 -> `快攻连击`
   - `gain_mana` / `gain_stamina` / 减费效果多 -> `资源爆发`
   - `lifesteal` / `soul` 多 -> `吸血续航`
   - `damage_shred_armor` / execute / 重击卡多 -> `破甲压制`
4. **epitaph**：模板句拼接，不做 AI 写作
   - 例如：`“持匕而归，毒火缠身，于第二轮回折断了门前的影子。”`

这是最稳的“中继承”。

## 五、锻铸者残响：如何做到“像玩家但不是玩家”

### 设计原则

锻铸者残响 = `固定 Boss 骨架` + `武器人格模板` + `附魔修正包` + `流派修正包`

它不是自由生成 AI，也不是玩家复制体。

### 1）武器人格模板（决定基础人格）

建议按武器建立 5 套基础人格：

- **长剑人格**：均衡压迫、攻守穿插、稳定斩杀
- **法杖人格**：蓄势-爆发、回合递进、后段处决
- **匕首人格**：多段命中、持续压血、节奏快
- **战锤人格**：低频重击、护甲推进、强压制
- **长弓人格**：标记猎杀、弱化压制、节奏拉扯

每套人格只需要 4~5 回合循环，不需要思考，不需要抽牌。

### 2）附魔修正包（决定腐化被动）

示例：

- `flame`：攻击附带灼烧
- `soul`：终结技后回复生命 / 汲取
- `void`：攻击后附弱化 / 诅咒感
- `abyss`：半血以下触发侵蚀强化
- `thunder`：多段攻击的尾击更重
- `bless`：定期获得护甲或净化减益
- `frost`：一期可先落为“迟滞/弱化”表达，不强求完整冻结敌人逻辑

### 3）流派修正包（决定两个技能模块）

从摘要里选 2 个标签，映射到模块：

- `战技爆发` -> 插入重击回合
- `法术爆发` -> 插入蓄势后爆发回合
- `灼烧` -> 攻击追加 burn
- `中毒` -> 多段附毒
- `高护甲` -> 防御回合更厚 / 开场护甲
- `蓄能` -> 每 2 回合获得强化，下一击更大
- `快攻连击` -> 开场多段快刀
- `资源爆发` -> 固定一回合强 buff，下一回合超额攻击
- `吸血续航` -> 终结回合回复
- `破甲压制` -> 重击附易伤或护甲粉碎

### 4）实现落点建议

为避免重写战斗框架，建议只做一个**很小的敌人被动层**，例如新增可选 `EnemyPassive[]`：

- `on_attack_apply_burn`
- `on_attack_apply_poison`
- `turn_start_gain_armor`
- `turn_start_gain_strength_every`
- `lifesteal_on_attack`
- `phase_shift_gain_armor_strength`

这样：

- 敌人仍然按固定 intents 走
- 只是额外挂少量可解释被动
- 非隐藏线敌人不需要改行为

### 5）为什么这比“完整镜像”更好

因为玩家的感受重点不是“它真的在打牌”，而是：

- 这像某个真实打过这里的人
- 它的武器、倾向、腐化味道都能看出来
- 但它已经不是人了，而是门留下的残响

这就够了。

## 六、裂隙之刃设计

### 定位

裂隙之刃不是普通毕业武器，而是**轮回链象征奖励 / 隐藏起始武器**。

### 基础设定

- 普攻：`5` 伤
- 被动：每场战斗开始时，从 `长剑 / 法杖 / 匕首 / 战锤 / 弓` 中随机继承一种基础武器人格，持续本场战斗

### 平衡原则

为了不压死普通路线，建议这样做：

1. **不进入普通锻造树**
   - 不出现在普通锻造配方里
   - 不从普通掉落池中出现
2. **人格继承的是“风格”，不是整把成品武器数值”**
   - 例如继承长剑人格，只拿“下一张战技减费”或“稳定重击倾向”
   - 不复制成钢武器、传说武器的完整数值上限
3. **每场战斗随机，不稳定但有戏剧性**
   - 强调叙事适应性，而不是线性最优解
4. **可作为隐藏起始武器，但不自动优于所有起始武器**
   - 它的价值在“可塑性”与“秘密感”，不是裸面板压制

### 特殊交互

如果上一位通关者使用的是裂隙之刃：

- 对应的锻铸者残响可以进入“人格摆动”版本
- 例如半血时从一种武器人格切换到另一种
- 只在残响 Boss 身上做，不让玩家这边也变复杂

## 七、深渊回响·N 的成长规则

你明确说了：不能只做“每层血量 +25”。

推荐成长结构：

### 1）保留少量基础成长

- 锻铸者残响：每层 `HP +8%`
- 真身 Boss：每层 `HP +6%`
- 每 2 层额外获得少量伤害加成或护甲加成

### 2）引入“回响词缀 / 变异规则”

每层从词缀池中选择 1 个；较高层可叠 2 个。

建议词缀池优先使用**当前系统容易表达的规则**：

- `灼蚀`：所有攻击额外施加 `burn`
- `淬毒`：多段攻击附带 `poison`
- `铁幕`：开场获得高额护甲，阶段切换再补一次
- `蓄压`：每 2 回合获得 `strength`
- `噬生`：造成伤害后回复少量生命
- `裂甲`：重击附带易伤 / 护甲粉碎感
- `影袭`：第一轮额外偏进攻，缩短观察窗口
- `容器筛选`：真身 Boss 在半血时清除减益并进入强化阶段

### 3）每层读取上一层通关摘要

- `第二轮回·侵影`：读取“第一轮回隐藏 Boss 首通者摘要”
- `深渊回响·1`：读取“第二轮回·侵影通关者摘要”
- `深渊回响·2`：读取“深渊回响·1 通关者摘要”
- 以此类推

### 4）没有真实摘要时的补位策略

内置 5~8 个“预设模板残响”：

- 烈焰匕首型
- 深渊法杖型
- 重甲战锤型
- 长剑破甲型
- 长弓压制型

这样即使玩家是第一次在某层测试，也不会断链。

## 九、持久化建议

### 建议拆成三层

#### A. `MetaProfile`：长期账户级进度

建议新增：

```ts
interface SecretCycleProgress {
  highestUnlockedTier: number
  highestClearedTier: number
  hiddenBossClearCount: number
  secretEntrySeenCount: number
  unlockedTitles: string[]
  unlockedStarterWeapons: string[]
  latestSummaryByTier: Record<string, ChampionSummary>
}
```

用途：
- 决定标题页解锁情况
- 决定首次/再次/多次进入隐藏线的文案版本
- 决定下一层该读取哪一份摘要

#### B. `GameState`：标题页临时选择

建议新增：

```ts
selectedCycleTier: number
```

用途：
- 标题页当前高亮项
- 不一定需要长期保存；如果以后想记住偏好，再挪到 settings

#### C. `RunState`：本局锁定层级

建议新增：

```ts
cycleTier: number
secretFlags?: {
  hiddenRouteEntered: boolean
  pendingSecretStage: 'none' | 'echo' | 'true_boss'
}
```

用途：
- 本局强绑定层级
- 存档继续时保持一致
- 后续如果战斗中断或中途保存，能恢复隐藏线阶段

### 为什么不是全放一个地方

- 轮回解锁显然是长期进度，必须在 `metaProfile`
- 当前标题页选择不是长期世界状态，没必要污染 meta
- 当前 run 的层级必须跟存档一起走，否则“继续冒险”会乱

这三个边界最清晰。

## 十、实现坑点与规避建议

### 1. 把“轮回解锁”和“当前 run 层级”混在一起

**风险**：继续旧档时层级被标题页新选择污染。  
**规避**：解锁进度放 meta，本局层级放 run。

### 2. 试图做完整玩家镜像

**风险**：会迅速变成一套新战斗系统。  
**规避**：严格限制到摘要 + 模板 + 模块。

### 3. 通关者摘要过度依赖战斗日志

**风险**：日志不稳定、维护成本高。  
**规避**：一期只扫最终 deck、武器、附魔。

### 4. 动态 Boss 数据无法存档恢复

**风险**：战斗中途保存再读档时，残响 Boss 无法重建。  
**规避**：不要用内存态临时注册；用 `cycleTier + summary` 可重建的确定性规格，或把必要 spec 写入 run。

### 5. 裂隙之刃做成新最强毕业武器

**风险**：普通路线失去意义。  
**规避**：强调随机人格与叙事价值，不给稳定上限。

### 6. 深渊回响只做数值膨胀

**风险**：玩起来像“换皮血牛”。  
**规避**：每层至少有一个回响词缀或阶段规则变化。

### 7. 隐藏线事件过长、测试成本高

**风险**：每次都要完整跑图验证。  
**规避**：保留现有快速测试入口思路，只给 `毁天灭地·TEST` 卡，方便验证流程而不破坏完整体验。

### 8. UI 一口气做太多污染特效

**风险**：成本高且容易破坏现有样式。  
**规避**：一期先做“多段文案 + 深色主题 + 问号污染 + 按钮文案更世界观化”。

## 十二、最终建议

如果按“世界观统一、实现可控、方便继续扩写”的标准看，我的建议非常明确：

- **不要做完整镜像**
- **不要做隐藏第四幕地图**
- **就做摘要驱动的双战秘密链**

最优落点是：

1. `metaProfile` 管解锁与摘要
2. `RunState` 管本局层级
3. `secret-cycle` 模块管流程、Boss 生成、文案
4. `main.ts` 只做串联

这样既保住“上一位冒险者黑化”的核心味道，又不会把当前 V3.0 分支拖进不可维护的复杂度里。
