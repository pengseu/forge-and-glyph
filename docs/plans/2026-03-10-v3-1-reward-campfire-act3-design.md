# V3.1 奖励提示、篝火、第三幕与玩家状态设计

## 目标

收口当前 `V3.1` 里 4 类体验问题：

- 获得奖励后没有明确提示，直接返回地图或推进流程。
- 篝火页需要整体参考背包页的结构与视觉。
- 第三幕战斗素材层导致英雄与敌人不清晰，甚至被埋住。
- 玩家 `buff / debuff` 呈现位置混乱，需要统一主次。

## 问题定位

### 1. 奖励提示缺口

- 宝库节点在 `src/main.ts` 中直接结算金币与材料后返回地图，没有 UI 摘要。
- 篝火回血、升级卡牌、升级武器同样直接回地图，只有日志，没有玩家可见提示。
- 事件系统已支持 `uiNotice`，但只有部分“得卡”事件接入。大量金币、材料、回血、属性变化类事件仍然没有摘要，因此会静默离场。
- 幕间奖励也存在类似问题：奖励被应用了，但没有统一的可见结算摘要。

### 2. 篝火样式落后

- `src/ui/scenes/campfire.ts` 仍以独立的小场景思路组织，和背包页的“主面板 + 标题 + 内容区 + 尾部统计/动作区”语言不一致。
- `src/ui/styles/batch3-core.css` 中篝火区域是单独的一套较轻结构，升级卡列表也没有复用背包卡片展示层级。

### 3. 第三幕战斗层问题

- 当前第三幕仅使用 `act3` 的通用层滤镜，缺少稳定的角色/敌人/前景层专用布局。
- 暗化滤镜与前景遮罩叠加后，角色主体识别度下降。
- 第二幕已经通过“共享稳定布局体系”的方式回退到可用状态，第三幕应采取相同思路，但再加上专属素材层修正。

### 4. 玩家状态区混乱

- 现在顶部 HUD 已显示部分状态，玩家角色附近也会显示状态，主次不清。
- 危险状态、功能状态、临场反馈混在一起，用户很难快速定位关键读数。

## 设计方案

### 1. 建立统一的“非战斗奖励摘要”层

- 扩展当前事件 `uiNotice` 思路，抽出统一的奖励摘要构建逻辑。
- 摘要覆盖以下类型：
  - 金币变化
  - 材料获得/消耗
  - 回血/回满
  - 卡牌获得/移除/升级/变换
  - 武器升级
  - 属性永久变化（力量、智慧、最大法力等）
- 使用位置：
  - 宝库节点
  - 篝火结算
  - 事件分支
  - 幕间奖励
- 展现原则：
  - 不做额外确认弹窗
  - 直接在当前场景内显示轻提示，再自动离开或推进
  - 若当前流程无法停留原场景，则在转场页或过渡态短暂显示摘要

### 2. 篝火整体向背包页靠拢

- 不是只改升级卡区域，而是让整个篝火页采用类似背包的面板式结构：
  - 头部：标题、副标题、关闭/返回语义按钮
  - 主区：左/中主操作区，右侧说明或状态区
  - 卡牌升级视图：直接复用背包卡片栈的视觉层次与网格节奏
  - 底部：当前生命、节点说明或操作返回区
- 菜单态与升级态保持同一视觉骨架，只切换主内容。
- 尽量复用背包已有 CSS token 和组件类名语义，不再为篝火单独发明一整套视觉体系。

### 3. 第三幕战斗的专属层级修正

- 保留第三幕背景素材，但不再只靠全局滤镜处理。
- 为 `scene-battle--act3` 增加独立舞台规则：
  - 玩家层和敌人层提升到可见优先级
  - 前景遮罩透明度降低，必要时缩窄覆盖区域
  - 第三幕角色和敌人的竖向位置上提，避免落进前景暗区
  - 若需，第三幕共享第二幕/第一幕稳定的单位定位体系，再只保留自己的背景与前景素材
- 目标是优先恢复“看得见人”和“能清楚分辨单位”，而不是先追求风格滤镜。

### 4. 玩家状态区重排

- 顶部 HUD 成为玩家状态的唯一完整信息源。
- 顶部状态分三层：
  - 危险 `debuff`：中毒、虚弱、易伤等，放在最醒目的主栏
  - 关键决策 `buff`：力量、智慧、蓄能、屏障等，放在次级状态栏
  - 附加说明：试炼、附魔触发、帮助按钮等，继续留在 meta 区
- 玩家身边保留弱化版“状态回显”：
  - 只保留少量小图标
  - 不承担完整阅读职责
  - 优先显示危险状态或最关键增益
- 原则：HUD 全量，角色旁边只做临场反馈，不再双份完整展示。

## 数据流

### 奖励摘要

1. 奖励逻辑产出变更结果。
2. 奖励摘要构建器把变更结果转成短文案列表。
3. 当前控制器决定摘要挂在哪个场景：
  - 事件页
  - 篝火页
  - 幕间页
  - 宝库过渡页或地图前轻提示
4. 场景显示摘要后自动离开或继续流程。

### 玩家状态

1. 战斗层先整理玩家所有状态源。
2. 按危险、关键、普通分类。
3. HUD 渲染完整状态组。
4. 玩家身边仅渲染受限回显集。

## 边界与约束

- 这次不做全局 toast 系统。
- 不重写整个战斗场景 DOM，只补战斗根类、状态区组织和第三幕 CSS。
- 奖励摘要必须尽量短，避免整段文案挡住核心 UI。
- 若同一结算包含多项收益，优先按“最重要变化 + 关键补充”组合，不追求逐项铺满。

## 测试策略

- `src/game/__tests__/events.test.ts`
  - 补金币/材料/回血类事件的 `uiNotice` 或摘要断言。
- `src/ui/scenes/__tests__/event-unit.test.ts`
  - 校验多类型奖励摘要渲染。
- `src/ui/scenes/__tests__/campfire-ui-unit.test.ts`
  - 补篝火新结构、菜单与升级视图的 HTML 断言。
- `src/ui/scenes/__tests__/battle-hud-unit.test.ts`
  - 校验玩家状态分层与第三幕根类/状态布局辅助函数。
- 必要时补一个针对宝库/幕间摘要的控制器测试或最小集成断言。

## 不做的事

- 不在这次引入新的全局通知中心。
- 不重做商店、奖励页、事件页为统一外壳，只做必要复用。
- 不做第三幕整套美术资源替换，只修现有素材的可见性和层级。
